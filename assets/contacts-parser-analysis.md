# Анализ contacts-parser: почему не находит контакты и что внедрить

## 1. Текущая архитектура

Парсер реализует **waterfall** из ~21 этапа:

| Этап | Источник | Метод |
|------|----------|-------|
| 0 | ClickHouse | `companies_meta` → director, name |
| 1 | Checko | Прямой переход на страницу компании, regex по body |
| 2 | DuckDuckGo (компания) | `ИНН контакты` → regex по body |
| 3 | Официальный сайт | Парсинг найденного URL из DDG |
| 4–16 | DDG (директор) | 14 разных запросов site:vk.com, site:hh.ru и т.д. → **regex только по странице DDG** |
| 18–21 | DDG (реестры) | site:fedresurs.ru, pub.fsa.gov.ru и т.д. → regex по странице DDG |

---

## 2. Основная причина: парсим DDG, а не целевые страницы

### Проблема

Для VK, LinkedIn, HH.ru, госзакупок и т.п. используется **один механизм** `runDirectorSource`:

1. Открывается `https://html.duckduckgo.com/html/?q=site:vk.com "Иванов Иван"`
2. Берётся `page.innerText('body')` — текст страницы **выдачи поиска DDG**
3. Из текста извлекаются email и телефон через regex

DuckDuckGo показывает **только сниппеты** — ссылки и короткие фрагменты. Email и телефон:

- почти никогда не публикуются в сниппетах VK/LinkedIn;
- скрыты на страницах профилей (авторизация, защита от ботов);
- встречаются в основном на HH.ru, zakupki.gov.ru, kad.arbitr.ru и т.п., но **только на самих страницах**, а не в сниппетах DDG.

Итог: парсинг только страницы DDG по сути не даёт контактов директора.

### Что реально работает сейчас

| Источник | Причина |
|----------|---------|
| Checko | Прямой переход на страницу компании — контакты на самой странице |
| DDG по компании | Редко, но иногда email/телефон есть в сниппетах |
| Официальный сайт | Прямой переход — контакты на странице |
| ЕФРСБ, Росаккредитация, ККТ | DDG сниппеты по реестрам — иногда есть контакты |

---

## 3. Что можно внедрить

### A. Переход на целевые страницы из DDG (важно)

Вместо парсинга только текста DDG:

1. Парсить ссылки из `.result__url` или `.links_main a`.
2. Открывать **первые 3–5 URL** (кроме checko, rusprofile, sbis).
3. С каждой целевой страницы извлекать email/телефон.

Риски: CAPTCHA, блокировки, ToS. Рекомендация:  
— маленькая задержка между запросами (1–2 с);  
— ограничить число переходов на домен (1–2 в минуту).

### B. API и сервисы с контактами

| Сервис | Что даёт | Интеграция |
|--------|----------|------------|
| **2GIS Places API** | Телефон, адрес, сайт организаций по ИНН/названию | `places/byid` по ИНН, нужен API‑ключ |
| **DaData findById/party** | ФИО руководителя, адрес, реквизиты (телефон/email — в премиум, уточнять) | REST API по ИНН |
| **Seldon.Basis** | Телефоны и email организаций | API, платный |
| **ofdata.ru** | Контракты госзакупок с контактами по ИНН | REST API, API‑ключ |
| **Zakupki360 API** | Закупки, компании, иногда контакты | Ограничения по лимитам |
| **SBIS/Saby API** | Контакты, ФИО, связи по ИНН | Платный (по ИНН) |
| **СПАРК‑Интерфакс** | Детали по компаниям, в т.ч. руководители | API, платный |

### C. Дополнительные поисковые запросы

Сместить акцент на страницы, где контакты часто открыто публикуются:

- `"ФИО" "телефон" OR "email" OR "контакт" суд арбитраж`
- `"ФИО" контактное лицо госзакупки`
- `"ФИО" ответственный исполнитель ИНН`
- `"ФИО" site:hh.ru резюме контакт`
- `"ФИО" site:list-org.com`
- `"ФИО" site:datalance.ru`
- `"ФИО" site:kartoteka.ru`
- `"ФИО" site:companycards.ru`

### D. Прямые переходы на страницы реестров

Вместо поиска через DDG — идти сразу на:

- `https://zakupki.gov.ru/epz/order/extendedsearch/search.html` — поиск по ИНН → переход к контракту → парсинг контактов;
- `https://kad.arbitr.ru/` — поиск по ФИО;
- `https://fedresurs.ru` — поиск по ИНН;
- страницы HH.ru с вакансиями/контактами работодателя (если есть).

### E. Расширение regex и форматов

- Дополнить `PHONE_REGEX` форматами:  
  `(XXX) XXX-XX-XX`, `8 (XXX) XXX-XX-XX`, международные форматы.
- Учитывать варианты email в сниппетах:  
  `[at]`, `(at)`, `(a)`, `@` в base64/обфускациях (с осторожностью).

### F. Альтернативные поисковики

- **Yandex** (`yandex.ru/search`) — больше локальных источников.
- **Google** — через SerpAPI/SearchAPI (платно).

## 5. Ошибка в коде

В `addDirectorContacts` (строка ~186) возможна опечатка в условии фильтра телефонов:

```typescript
phones.filter(p => p.replace(/[^0-9]/g, '').length >= 10).forEach(...)
```

Синтаксис корректен, но стоит добавить отсечение заведомо невалидных номеров (например, служебных, коротких).

---

## 6. Резюме

Парсер почти не находит личные контакты директора, потому что:

1. VK, LinkedIn, Telegram и т.п. ищутся через DDG, но парсится только страница поиска, а не целевые страницы.
2. Контакты чаще всего лежат на страницах HH.ru, zakupki.gov.ru, арбитража и реестров, а не в сниппетах DDG.
3. Не используются API 2GIS, DaData, ofdata.ru и др., которые дают готовые контакты.

Оптимальная последовательность:

1. Реализовать переход по ссылкам из DDG и парсинг целевых страниц.
2. Подключить 2GIS API (если есть ключ).
3. Подключить API госзакупок (ofdata.ru или Zakupki360).
4. Обогатить поисковые запросы и добавить прямые переходы на реестры, где это возможно.
