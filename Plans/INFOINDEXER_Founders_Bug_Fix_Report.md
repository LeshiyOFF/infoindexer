## Введение: что такое INFOINDEXER

**INFOINDEXER** — это OSINT-система (Open Source Intelligence), которая собирает и анализирует публичные данные о российских компаниях и их связях.

### Что она делает?

Представьте, что вы хотите узнать:
- Кто владеет конкретной компанией?
- Какие компании связаны между собой через общих директоров?
- Есть ли у компании санкции или судимости?

INFOINDEXER отвечает на эти вопросы, собирая данные из множества источников:
- **ЕГРЮЛ / ЕГРИП** — государственные реестры компаний РФ
- **OpenSanctions** — международная база санкций
- **ФНС** — налоговая служба (60M+ финансовых отчётов)
- И ещё 20+ источников

### Почему это важно?

- Журналисты находят схемы отмывания денег
- Банки проверяют контрагентов
- Следователи раскрывают финансовые преступления
- Бизнес анализирует конкурентов

## В чём заключалась проблема

### Описание симптома

В системе INFOINDEXER при просмотре карточки компании раздел **"Учредители" (Founders)** отображался пустым, несмотря на то, что данные об учредителях содержались в исходном источнике — выгрузке ЕГРЮЛ.

### Простая аналогия

Представьте, что у вас есть телефонная книга с двумя списками:

```
Список 1 (люди):
├── 9a3f2b1c... → Иванов Иван Иванович
├── 8c2e1a3d... → Петров Пётр Петрович
└── ...

Список 2 (кто кем владеет):
├── ООО "Ромашка" → владелец: ru-inn-123456789012
├── ООО "Лютик" → владелец: ru-inn-987654321098
└── ...
```

Проблема: система не понимала, что `ru-inn-123456789012` из второго списка — это то же самое, что `9a3f2b1c...` из первого списка.

### Визуальное представление

**Было (до исправления):**

```
┌─────────────────────────────────────────┐
│ ООО "РОМАШКА"                           │
│                                         │
│ ИНН: 1234567890                         │
│ Директор: Иванов И.И.                   │
│                                         │
│ Учредители:                             │
│   [нет данных]                          │  ← ПРОБЛЕМА
└─────────────────────────────────────────┘
```

**Ожидалось (стало):**

```
┌─────────────────────────────────────────┐
│ ООО "РОМАШКА"                           │
│                                         │
│ ИНН: 1234567890                         │
│ Директор: Иванов И.И.                   │
│                                         │
│ Учредители:                             │
│ • Иванов Иван Иванович — 50%            │
│ • Петров Пётр Петрович — 30%            │
│ • Сидоров Сидор Сидорович — 20%         │
└─────────────────────────────────────────┘
```

### Влияние на систему (почему это критично)

- **Потеря данных:** Информация о реальных владельцах компаний была недоступна для пользователей
- **Неполный анализ:** OSINT-исследования не могли учитывать структуру собственности — это значит, что схемы владения компаниями оставались скрытыми
- **Неверные связи:** Графы связей между компаниями и людьми были неполными — усложняет выявление бенефициаров
- **Финансовые потери:** Бизнес не мог полноценно проверять контрагентов перед сделками

## Корневая причина простыми словами

### Что произошло?

Система получает данные от OpenSanctions. Эти данные структурированы особым образом:

1. **Каждый человек** имеет свой уникальный идентификатор (UUID)
2. **Каждая компания** тоже имеет свой UUID
3. **Связи** между ними (кто кем владеет) используют ИНН как идентификатор

### Почему это проблема?

Потому что компьютер **буквально не понимает**, что это один и тот же человек.

Для компьютера:
```
"ru-inn-123456789012" ≠ "9a3f2b1c-5e4d-4a8f-9c7b-1e2d3f4a5b6c"
```

Это как если бы у вас было две записи об одном человеке:
- Запись 1: "Иванов И.И., паспорт 1234 567890"
- Запись 2: "Иванов Иван Иванович"

И вы не поняли бы, что это один человек.

## Решение Phase A: Identity Mapping

### Идея решения

Нужна была **таблица-переводчик**, которая скажет системе: "Запись с ИНН `123...` и запись с UUID `9a3f...` — это один и тот же человек".

### Простая аналогия

Представьте, что у вас есть справочник:

```
┌───────────────────┬──────────────────────┐
│ Как написано      │ Кто это на самом     │
│ в связях          │ деле в базе людей     │
├───────────────────┼──────────────────────┤
│ ru-inn-123...     │ Иванов Иван И.       │
│ ru-inn-456...     │ Петров Пётр П.       │
│ 9a3f2b1c...       │ Иванов Иван И.       │
└───────────────────┴──────────────────────┘
```

Теперь система может "перевести" ИНН в UUID и найти нужного человека.

### Как это работает технически

Была создана специальная таблица `egrul_identity_mapping`:

```
Таблица egrul_identity_mapping:
┌────────────────────┬──────────────┬─────────────────────┐
│ raw_id             │ canonical_id │ Что это значит      │
├────────────────────┼──────────────┼─────────────────────┤
│ ru-inn-12345678... │ 9a3f2b...    │ ИНН → UUID человека │
│ 9a3f2b1c...        │ 9a3f2b...    │ UUID → тот же UUID  │
│ ru-ogrn-1234567... │ 8c2e1a...    │ ОГРН → UUID компании│
└────────────────────┴──────────────┴─────────────────────┘
```

Теперь запрос к базе данных работает так:

```
1. Берём ИНН из связи: ru-inn-123456789012
2. Ищём в таблице маппинга: ah, это UUID 9a3f2b1c...
3. Берём данные человека по этому UUID
4. Показываем пользователю
```

### Что было создано

| Компонент | Что делает |
|-----------|------------|
| `IdentityMappingService` | Строит таблицу-переводчик автоматически |
| `CompanyMergerService` | Объединяет данные о директорах и учредителях |
| `EntityParserService` | Разбирает данные из OpenSanctions |

## Решение Phase B: External Enrichment

### Зачем понадобилась Phase B?

Phase A отлично работает, когда ИНН человека явно указан в данных. Но бывают случаи:

1. **ИНН отсутствует** в записи о человеке
2. **Разные форматы** записи (с пробелами, без, с дефисами)
3. **Опечатки** в данных
4. **Человека нет** в базе OpenSanctions, но он есть в ЕГРЮЛ

В таких случаях даже таблица-переводчик не помогает.

### Идея решения

Если мы не можем найти связь автоматически — давайте **спросим у внешнего сервиса**.

**DaData** — это сервис, который по ИНН может найти ФИО человека в государственных реестрах России.

### Алгоритм обогащения (по шагам)

```
┌─────────────────────────────────────────────────────────────┐
│ ШАГ 1: Находим "потерянные" ИНН                            │
│                                                             │
│ Ищем ИНН, которые есть в связях, но нет в таблице маппинга  │
│ Результат: 10,000 "потерянных" ИНН                         │
├─────────────────────────────────────────────────────────────┤
│ ШАГ 2: Спрашиваем DaData                                    │
│                                                             │
│ Отправляем ИНН: 111222333444                               │
│ DaData отвечает: Иванов Иван Иванович                      │
├─────────────────────────────────────────────────────────────┤
│ ШАГ 3: Ищем похожее ФИО в нашей базе                       │
│                                                             │
│ DaData сказал: "Иванов Иван Иванович"                     │
│ Ищем в базе:                                                │
│   - "Иванов Иван Иванович" ✓ (нашли!)                      │
│   - "Иванов И.И." ✓ (похоже!)                              │
│   - "Ivanov Ivan Ivanovich" ✓ (транслит!)                  │
├─────────────────────────────────────────────────────────────┤
│ ШАГ 4: Создаём связь                                        │
│                                                             │
│ ru-inn-111... ↔ UUID_ИвановаИвановича                      │
│ Confidence: 95% (мы уверены на 95%)                        │
└─────────────────────────────────────────────────────────────┘
```

### Что такое Fuzzy Matching?

**Fuzzy Matching** — это алгоритм, который понимает, что строки похожи, но не идентичны.

**Примеры:**

| Сравниваем | Результат | Почему |
|------------|-----------|---------|
| "Иванов Иван Иванович" vs "Иванов Иван Иванович" | Идеал (100%) | Точно совпадает |
| "Иванов Иван Иванович" vs "Иванов И.И." | Хорошо (70%) | Инициалы |
| "Иванов Иван Иванович" vs "Ivanov Ivan Ivanovich" | Плохо (40%) | Разный алфавит |
| "Иванов Иван Иванович" vs "Петров Пётр Петрович" | Нет (0%) | Разные люди |

Используется **алгоритм Левенштейна** — он считает минимальное количество изменений (вставок, удалений, замен), чтобы превратить одну строку в другую.

```
"Иванов" → "Иванов"     = 0 изменений (100% совпадение)
"Иванов" → "Иваноф"     = 1 замена   (80% совпадение)
"Иванов" → "Иван"       = 2 удаления  (60% совпадение)
```

### Компоненты Phase B

| Компонент | Что делает |
|-----------|------------|
| `DaDataAdapter` | Общается с DaData API |
| `FuzzyMatcherService` | Сравнивает ФИО нечётко |
| `ExternalEnrichmentService` | Координирует весь процесс |
| `EnrichmentWorker` | Работает в фоне, не блокируя систему |

## Архитектурные улучшения

Помимо исправления самого бага, код был полностью переработан.

### Что было неправильно?

Раньше весь код находился в одном большом файле (~573 строки). Это как если бы вы сложили все инструменты в один ящик — и хлам, и ценное, и то что нужно каждый день, и то что нужно раз в год.

```
egrul-sync.service.ts (573 строки):
  ├── Загрузка данных из интернета
  ├── Чтение JSON
  ├── Сохранение в базу данных
  ├── Обработка ошибок
  ├── Отчётность о прогрессе
  ├── Логика маппинга
  └── Всё вперемешку...
```

### Что стало правильным?

Код был разбит на **30 отдельных файлов** по принципу: **один класс = одна задача**.

```
src/core/
├── entities/           -- Описания структур данных
├── parsers/            -- Разбор специфичных форматов
├── infrastructure/     -- Работа с внешними системами (HTTP, Redis)
├── repositories/       -- Работа с базой данных
├── services/           -- Бизнес-логика
├── adapters/           -- Интеграция с внешними API
└── workers/            -- Фоновые процессы
```

### Почему это важно?

| Аспект | Было | Стало |
|--------|------|-------|
| Найти ошибку | Перебрать 573 строки | Знать где искать (один файл) |
| Добавить функцию | Риск сломать всё | Добавить один файл |
| Тестировать | Нужно всё | Тестировать только кусок |
| Понять код | Недели | Минуты |

### Применённые принципы (SOLID)

Это аббревиатура из 5 принципов хорошего кода:

| Принцип | Простыми словами |
|---------|-----------------|
| **S**ingle Responsibility | Каждый класс делает одну вещь хорошо |
| **O**pen/Closed | Можно расширять, не меняя старый код |
| **L**iskov Substitution | Детали можно заменить без поломки |
| **I**nterface Segregation | Минимум зависимостей |
| **D**ependency Inversion | Зависимости не привязаны к реализации |
